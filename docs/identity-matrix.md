## Why managed identity beats SP + secret for storage access (FR-2.3)

The classic pattern (SP with a client secret in a Databricks secret scope, `fs.azure.account.oauth2...` cluster configs) has four structural problems:

1. **A secret exists** — it can be exfiltrated from configs, logs, or a notebook (`dbutils.secrets.get` redaction is best-effort).
2. **It expires** — someone owns rotation, and pipelines break when they forget.
3. **It is workspace-wide by default** — cluster-level OAuth configs bypass table-level governance.
4. **It is invisible to UC lineage/audit** — access happens as an opaque app id.

The Access Connector MI inverts all four: no secret material exists anywhere; Azure mints short-lived tokens on demand; UC decides *per table/volume* whether a given principal may use the credential; every access is attributable in UC audit logs. This is why FR-2.3 routes **all** data-plane storage access through the MI, while SQL (which has no MI-based JDBC path from Spark without extra machinery) keeps credentials in Key Vault — see [ADR-0003](adr/0003-jdbc-vs-lakehouse-federation.md) / FR-2.6 for the trade-off and the Lakehouse Federation alternative.

## Credential-flow summary

```
Azure DevOps pipeline ──OIDC token exchange──> sp-lakeforge-infra ──ARM──> Terraform
Azure DevOps pipeline ──OIDC token exchange──> sp-lakeforge-deploy ──Databricks API──> bundle deploy
Cluster (UC table/volume I/O) ──UC storage credential──> Access Connector MI ──token──> ADLS
Cluster (JDBC ingest) ──KV-backed secret scope──> Key Vault ──secret──> Azure SQL
Negative test ──KV secret (client id+secret)──> sp-lakeforge-analyst ──Entra token──> Databricks
```

Zero secrets in Git, bundle files, or pipeline variables (NFR-1). The only stored secrets anywhere are: SQL admin credentials and the analyst test secret — both exclusively in Key Vault, both created and rotated by Terraform.
